# Contributing to Project Phoenix

Thank you for your interest in contributing to Project Phoenix! This document provides guidelines for different types of contributions.

---

## Ways to Contribute

### 1. Share Your Implementation

Have you applied the Phoenix pipeline to a legacy modernization project? We'd love to hear about it.

**What to include:**
- Legacy stack and codebase characteristics
- Which agents you used and how
- Timeline and team composition
- What worked well and what didn't
- Business rules count and coverage achieved

**How to share:**
- Open an issue with the label `implementation-showcase`
- Include enough detail for others to learn from your experience

---

### 2. Report Issues or Suggest Improvements

Found something unclear, incorrect, or missing?

**Before submitting:**
- Check existing issues to avoid duplicates
- Provide specific examples when possible
- Reference specific files and sections

**Issue labels:**
- `documentation`: Unclear or incorrect documentation
- `question`: Need clarification on methodology concepts
- `enhancement`: Suggestion for improvement
- `validation-data`: Challenges or questions about methodology claims

---

### 3. Contribute Documentation Improvements

**Types of documentation contributions:**
- Clarifying existing explanations
- Adding examples or use cases
- Fixing typos or formatting
- Improving diagrams or visual aids

**Process:**
1. Fork the repository
2. Make your changes
3. Submit a pull request with:
   - Clear description of what you changed and why
   - Reference to any related issues
   - Confirmation that you've reviewed the Design Principles below

---

### 4. Contribute Validation Data

Have you tested the Phoenix methodology in your environment?

**We welcome:**
- Case studies from Phoenix engagements
- Comparative data (Phoenix vs. traditional migration)
- Agent output quality assessments
- Cross-domain application reports

**Requirements:**
- Clear methodology description
- Observable, measurable results
- Honest reporting (including negative results)
- Sufficient detail for reproducibility

---

## Design Principles for Contributors

When contributing, please respect these core methodology principles:

### 1. Intent Over Implementation
Every contribution should focus on what systems do, not how they're coded. Phoenix extracts business intent, not code structure.

### 2. Human-in-the-Loop
No proposed change should eliminate human decision points. The gate model is fundamental to Phoenix's risk management.

### 3. Greenfield Commitment
Phoenix does not produce legacy-style output. Contributions should reinforce the greenfield approach, not introduce translation patterns.

### 4. Observable Validation
Claims about methodology effectiveness should be grounded in measurable outcomes -- coverage percentages, timeline data, team sizes.

### 5. Honest Reporting
Report failures as clearly as successes. The methodology improves through honest assessment, not selective evidence.

## Pull Request Guidelines

### Documentation Changes

**Format:**
- Use Markdown formatting consistent with existing docs
- Maintain the existing structure and hierarchy
- Use backticks for code and technical terms
- Keep language clear and concise

**Before submitting:**
- Read through your changes for clarity
- Check links to ensure they work
- Verify any formulas or technical claims
- Run a spell-checker

---

## Review Process

1. **Initial review** (1-3 days): Maintainer reviews for alignment with methodology principles
2. **Feedback**: Requests for changes or clarifications if needed
3. **Approval**: Once aligned with principles and quality standards
4. **Merge**: Changes incorporated into main branch

## Code of Conduct

### Our Standards

- **Respectful**: Treat all contributors with respect
- **Constructive**: Provide helpful, actionable feedback
- **Empirical**: Ground discussions in observable data
- **Honest**: Report both successes and failures
- **Open**: Welcome diverse perspectives and use cases

### Unacceptable Behavior

- Personal attacks or harassment
- Speculation presented as fact
- Dismissing empirical evidence without data
- Promotion of unrelated products or services

---

## Getting Started

Ready to contribute?

1. **Fork** the repository
2. **Create a branch** for your changes
3. **Make your edits** following these guidelines
4. **Submit a pull request** with clear description
5. **Respond to feedback** during review

Thank you for helping improve Project Phoenix!
